\subsection{Overview}\label{overview}}

The code in this replication package generates Tables~1 to 6 and Figures~1 to 2 in the paper titled ``Priced risk in corporate bonds,'' by Dickerson, Mueller, and Robotti (2023, henceforth DMR).
The code to generate Tables~1 to 5 and Figures~1 to 2 are written in Matlab.
Table~6 is generated with Python only.
The file \texttt{RunAllResults.m}, the `one-push-button', takes about 2 minutes to execute, and will produce \textit{all} of the results in the paper except for Table~6.
These results are printed to the Matlab console, saved as \texttt{AllTablesReport} and also stored in Matlab arrays such that they can be exported to other file formats.
The results for Table~6 are produced by executing the Python script \texttt{Table6.py}.
The authors have set up a dedicated replication/companion website (Open Source Bond Asset Pricing) which can be accessed \href{https://openbondassetpricing.com/}{here}.
This README file guides the user through replicating the results of the paper as well as constructing the main database from scratch.

\subsection{Instructions for Replicating the Main Analyses}

\begin{enumerate}
    \item Open Matlab, navigate to the DMR replication package folder, open the \texttt{RunAllResults.m} Matlab script file, and press \texttt{F5}, or click `Run.' The script should take $<$ 2 minutes to execute.
    \item The results for Tables~1-5 will be printed to screen (on the console). The results can be also be viewed in \texttt{.txt} format by opening the \texttt{AllTablesReport} file in Matlab itself. Figures~1-2 will be saved to file.
   \item Execute the Python script \texttt{Table6.py}. Change the file path first on line 54. Results are exported to Excel \texttt{.csv} files called Table6AB.csv and Table6CD.csv respectively.
\end{enumerate}

\subsection{Instructions for Data Preparation - List of Programs}
To generate the main bond-level panel and factors for analyses, run the following scripts in order from the subfolder DataPrep:

\begin{enumerate}
        \item Run \texttt{MakeIntra\_Daily.py}. This script outputs the daily bond-level panel with clean prices and volumes. Run on the \href{https://wrds-www.wharton.upenn.edu/pages/support/the-wrds-cloud/using-ssh-connect-wrds-cloud/}{WRDS cloud}.  Runtime is $\sim$4.50 hours.
        \item Run \texttt{MakeBondDailyMetrics.py}. This script outputs the daily bond accrued interest, dirty prices, duration, convexity, and yields. With 14 cores, runtime is $\sim$20 minutes.
        \item Run \texttt{MakeBondMonthlyMetrics.py}. This script outputs the monthly bond returns, excess returns, bond yields, duration, and convexity. Negligible runtime (under 1 minute).
        \item Run \texttt{MakeCreditSpreads.py}. This script estimates monthly bond credit spreads. With 14 cores, runtime is $\sim$20 minutes.
        \item Run \texttt{MakeIlliquidity.py}. This script estimates monthly bond illiquidity following Bao et al. (2011). Negligible runtime (under 5 minutes).
        \item Run \texttt{MakeRatings.py} and \texttt{MakeAmountOutstanding.py} in any order. This script downloads bond ratings and bond amount outstanding. Negligible runtime (under 1 minute).
        \item Run \texttt{MakeDataBaseTRACE.py}. This script downloads the data processed in the prior scripts and generates the final database. Negligible runtime (under 1 minute).
        \item Run \texttt{MakeBBW4Factors.py} to produce the factors.
        \item \texttt{MakePortfolioBasisAssets.py} contains the functions to produce the 32 portfolios.
        \item Run \texttt{CreateTableA1.py} to produce Table A1.
        \item Run \texttt{CreateTableA2.py} to produce Table A2.
    \end{enumerate}

\subsection{A note on the TRACE database}
Creating the TRACE database from scratch (from the intraday to the final monthly panel) is subject to various coding design choices and possible heterogeneity across programming languages. We have processed the data in several ways, with different choices of functions, and we consistently arrive at a similar-sized dataset with a set of results that are always closely aligned. We believe our choices (throughout the codebase) are easy to justify. We strongly urge researchers to use the publicly available WRDS database going forward to completely eliminate any possible heterogeneity from having to `self-process' the TRACE data from scratch. If you believe that the TRACE data preparation codebase can be improved, please contact: \texttt{a.dickerson@warwick.ac.uk}.
